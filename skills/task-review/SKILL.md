# Task Review — 진행사항 보고서 자동 생성

작업 보고용 문서를 자동 생성합니다. git log와 diff를 분석하여 설정된 출력 경로 하위에 **단일 파일**로 기록합니다.

## 설정

출력 경로는 `.claude/skill-config.yaml`에서 설정합니다. 설정이 없으면 기본값 `docs/task-reviews/`를 사용합니다.

```yaml
# .claude/skill-config.yaml
task_review:
  output_dir: "docs/task-reviews/"   # 기본값
```

## 산출물

**파일 1개만 생성:**

```
{output_dir}/{주제}-progress-{YYYYMMDD}.md
```

- 예시: `docs/task-reviews/product-planning-pipeline-progress-20260325.md`

## 실행 절차

### 1단계: 데이터 수집

#### 1-1. Git 이력 수집

아래 git 명령을 실행하여 정보를 수집합니다:

```bash
# 오늘 날짜의 커밋 목록 (작업자, 시간, 메시지)
git log --since="$(date +%Y-%m-%d) 00:00:00" --until="$(date -v+1d +%Y-%m-%d) 00:00:00" --pretty=format:"%h|%ai|%an|%s" --no-merges

# 오늘 변경된 파일 통계
git log --since="$(date +%Y-%m-%d) 00:00:00" --until="$(date -v+1d +%Y-%m-%d) 00:00:00" --pretty=format:"" --name-status --no-merges | sort -u

# 현재 브랜치
git branch --show-current

# 커밋되지 않은 변경사항 (특정 범위 필터링 시 경로 제한)
git status --short
```

인자로 날짜가 주어지면 (예: `/task-review 2026-03-24`) 해당 날짜 기준으로 수집합니다.
인자로 날짜 범위가 주어지면 (예: `/task-review 2026-03-20 2026-03-25`) 해당 기간 기준으로 수집합니다.
인자로 범위 키워드가 주어지면 (예: `/task-review 파이프라인 관련만`) 해당 범위의 파일/커밋만 필터링합니다.

#### 1-2. PIMS 시간 데이터 수집 (선택적)

PIMS 시간 기록 데이터를 수집하여 보고서에 병합합니다. PIMS 연동 실패 시 git 데이터만으로 보고서를 생성합니다.

```
1. 해당 기간의 시간 기록 조회:
   mcp__pims__get_time_entries(from={시작_날짜}, to={종료_날짜})

2. 내 담당 이슈 상태 조회:
   mcp__pims__get_my_issues()

3. 수집 항목:
   - 이슈별 투입 시간 집계
   - 활동 유형별 분류 (개발, 회의, 리뷰 등)
   - 시간 기록이 없는 커밋 감지 ("시간 미기록 커밋")

4. 미기록 커밋 감지:
   - git 커밋의 날짜/티켓번호와 PIMS 시간 기록을 교차 확인
   - 커밋은 있지만 해당 날짜에 PIMS 시간 기록이 없는 경우:
     "⚠️ 시간 미기록 커밋 {N}건 — `/log-work`로 기록하세요"

※ PIMS API 호출 실패 시: "PIMS 시간 데이터 조회 실패 — git 데이터만으로 보고서를 생성합니다." 경고 후 계속 진행
```

#### 1-3. 대화 맥락 수집

현재 대화 세션에서 주요 프롬프트 요청과 AI 작업 과정을 분석합니다:

- **사용자 요청**: 각 작업에 대한 사용자의 원본 요청 내용
- **AI 작업 과정**: 주요 판단 사항, 시행착오, 해결 방법
- **중요 결정**: 구조 변경, 기술 선택 등 주요 의사결정 내용

**수집 원칙:**
- 현재 대화 세션의 메시지 흐름을 시간 순으로 분석
- 커밋 시간대와 매칭하여 어떤 요청으로 어떤 커밋이 생성되었는지 연결
- 민감 정보(개인 정보, 인증 키 등)는 필터링하여 제외

### 2단계: 진행사항 보고서 생성

`{output_dir}/{주제}-progress-{YYYYMMDD}.md` 파일을 생성하거나 업데이트합니다.

## 파일 관리 규칙

### 저장 경로

- `{output_dir}/{보고서파일명}.md` (output_dir는 `.claude/skill-config.yaml`의 `task_review.output_dir`, 기본값: `docs/task-reviews/`)

### 파일 네이밍

- `{주제}-progress-{YYYYMMDD}.md`
- 예: `product-planning-pipeline-progress-20260325.md`

### 네이밍 규칙

- kebab-case 사용
- 날짜 접미사 필수 (생성일 기준)
- 영문 파일명 기본

### 필수 헤더

```markdown
# {보고서 제목}

> 생성일: {날짜}
> 최종 수정: {날짜}
> 작성자: {AI 도구명 또는 사용자}
> 대상: {관련 파이프라인/기능}

---
```

### 금지 사항

- 설정된 `output_dir` 외부에 보고서 파일 생성 금지
- `final`, `v2`, `review` 같은 임의 접미사로 상태 관리 금지
- README.md 임의 생성 금지

## 규칙

### 기본 규칙

- **산출물은 단일 파일** — 로그와 보고서를 분리하지 않고 하나의 파일에 통합
- 기존 파일이 있으면 덮어쓰지 않고 업데이트
- 커밋 메시지가 한국어이므로 요약도 한국어로 작성
- 커밋이 없으면 미커밋 변경사항만 기록하고 "커밋 필요" 알림 표시
- 범위 필터링 인자가 있으면 해당 범위의 파일/커밋만 포함

### 대화 맥락 포함 규칙

- **타임라인 섹션**: 각 작업 블록에 "사용자 요청", "AI 작업 과정", "커밋 내역" 순서로 기록
- **사용자 요청**: 원본 프롬프트를 인용문(> )으로 표시. 너무 길면 핵심만 요약
- **AI 작업 과정**: 주요 판단 사항, 시행착오, 해결 방법을 불릿 리스트로 정리
- **커밋과 대화 매칭**: 커밋 시간을 기준으로 해당 커밋을 만든 프롬프트를 역추적하여 연결
- **민감 정보 필터링**: 개인 정보, API 키, 인증 토큰 등은 `[FILTERED]`로 대체
- 대화 맥락이 없는 경우(과거 날짜 조회 등)에는 커밋 정보만 기록

### 섹션별 규칙

- **개선 전/후 비교**: 변경이 의미 있는 경우에만 포함. 수치 변화(%, 건수) 명시
- **후속 작업**: 완료된 항목은 취소선 + ✅, 미완료 항목은 우선순위와 함께 표시
- **결론**: 불릿 리스트로 성과 정리 + 다음 단계 명시
