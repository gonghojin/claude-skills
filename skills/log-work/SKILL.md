# log-work — 작업 시간 기록

PIMS 티켓에 **작업 시간을 기록**하고, 선택적으로 **진행률을 갱신**합니다.
중복 기록 감지, dry-run 모드를 지원합니다.

## Usage

```bash
/log-work #118746 2h              # 특정 티켓에 2시간 기록
/log-work 2h                      # git log에서 티켓 자동 감지 후 기록
/log-work meeting 1h              # 회의 시간 기록
/log-work --dry-run #118746 2h    # 실행 없이 미리보기만
```

**파라미터:**
- `#issue-number` — (선택) PIMS 티켓 번호. 생략 시 git log에서 자동 감지
- `<hours>` — 기록할 시간 (예: `2h`, `1.5h`, `30m`)
- `meeting` — 회의 시간 기록 모드 (log_meeting_time 사용)
- `--dry-run` — 실제 기록 없이 미리보기만 출력

**출력:**
- 시간 기록 결과 요약
- (선택) 진행률 갱신 결과

---

## Instructions

당신은 개발자의 시간 기록을 돕는 어시스턴트입니다.
PIMS에 정확하게 시간을 기록하고, 중복을 방지하며, 진행률 갱신까지 한 번에 처리합니다.

### 1단계: 티켓 결정

```
Case A: 티켓 번호가 명시된 경우
  mcp__pims__get_issue(issue_id={번호}, full_description=false)
  → 티켓 제목, 상태, 현재 진행률 확인

Case B: 티켓 번호 생략 시 — git log에서 자동 감지
  git log --since="today 00:00:00" --pretty=format:"%s" --no-merges
  → 커밋 메시지에서 #숫자 패턴 추출
  → 중복 제거 후 후보 목록 생성

  후보가 1개: 해당 티켓 자동 선택
  후보가 2개+: 사용자에게 선택 요청
    "오늘 작업한 티켓:
     1. #118746 KCM 에이전트 BE
     2. #118815 에이전트 상세 FE
     어느 티켓에 기록할까요?"
  후보가 0개: 
    mcp__pims__get_my_issues()로 내 진행중 이슈 목록 표시
    사용자에게 선택 요청

Case C: meeting 모드
  회의 시간 기록 — 티켓 선택 없이 바로 3단계(활동 유형)로 이동
```

### 2단계: 중복 기록 감지

```
mcp__pims__get_time_entries(issue_id={티켓번호}, from={오늘_날짜}, to={오늘_날짜})

Case: 오늘 이미 기록이 있는 경우
  "⚠️ 오늘 이 티켓에 이미 {기존시간}h ({활동유형}) 기록되어 있습니다."

  동일 시간 + 동일 활동 유형이면:
    "🔴 동일한 기록이 이미 있습니다. 중복일 수 있습니다.
     그래도 추가로 {N}h 기록할까요? (y/n)"

  다른 시간 또는 다른 활동 유형이면:
    "기존 {기존시간}h에 추가로 {N}h 기록합니다. 진행할까요?"

Case: 기록이 없는 경우
  → 다음 단계로 진행
```

### 3단계: 활동 유형 선택

```
mcp__pims__get_time_activities()

활동 유형 목록을 사용자에게 표시:
  "활동 유형을 선택해주세요:
   1. 개발 (R&D)
   2. 코드리뷰
   3. 회의
   4. 테스트/검증
   5. 자료작성
   6. 기타"

meeting 모드인 경우: 자동으로 "회의" 선택
이전 기록이 있고 동일 티켓이면: 이전 활동 유형을 기본값으로 제안
```

### 4단계: dry-run 출력 + 사용자 확인

```
기록 예정 내용을 표시합니다:

"📝 시간 기록 미리보기:
 - 티켓: #{번호} {제목}
 - 시간: {N}h
 - 활동: {활동유형}
 - 날짜: {오늘}
 - 코멘트: {자동 또는 사용자 입력}

 기록할까요?"

--dry-run 모드인 경우:
  위 출력 후 종료. 실제 API 호출 없음.
```

### 5단계: 시간 기록 (Retry-with-Confirmation)

```
Case: 일반 모드
  mcp__pims__log_time(
    issue_id={번호},
    hours={시간},
    activity_id={활동유형_id},
    comments="{코멘트}"
  )

Case: meeting 모드
  mcp__pims__log_meeting_time(
    hours={시간},
    comments="{코멘트}"
  )

성공 시: "✅ #{번호}에 {N}h ({활동유형}) 기록 완료"

실패 시:
  - 네트워크/타임아웃: "재시도할까요?" (최대 2회)
  - 권한 오류: "권한이 없습니다. 수동으로 처리하세요."
  - 데이터 오류: "입력값을 확인해주세요." + 수정 제안
  - 기타: 원본 에러 출력
```

### 6단계: 진행률 갱신 제안 (선택)

```
시간 기록 성공 후:

mcp__pims__get_issue(issue_id={번호})
→ 현재 진행률 확인

"현재 진행률: {N}%
 진행률을 업데이트할까요? (숫자 입력 또는 'n' 스킵)"

사용자가 숫자를 입력하면:
  mcp__pims__update_issue(issue_id={번호}, done_ratio={입력값})
  "✅ 진행률 {이전}% → {이후}% 갱신 완료"

사용자가 'n' 또는 스킵:
  진행률 변경 없이 종료
```

### 7단계: 결과 요약

```
"📋 기록 요약:
 - 티켓: #{번호} {제목}
 - 시간: {N}h ({활동유형})
 - 진행률: {이전}% → {이후}% (또는 '변경 없음')
 - 오늘 누적: {오늘_총_시간}h"
```

## 시간 파싱 규칙

```
입력 → 시간(hours)
2h   → 2.0
1.5h → 1.5
30m  → 0.5
1h30m → 1.5
90m  → 1.5
```

---

## 규칙

- PIMS 기록 전 반드시 dry-run 출력 + 사용자 확인 필요
- `--dry-run`에서는 PIMS 쓰기 호출 절대 없음
- 중복 기록 감지: 동일 티켓 + 동일 날짜 + 동일 시간 + 동일 활동 유형 시 강조 경고
- 진행률 갱신은 항상 선택적 (사용자가 원할 때만)
- meeting 모드에서는 활동 유형 자동 선택 (회의)
- PIMS에 데이터를 삭제하는 동작은 지원하지 않음
- 모든 텍스트는 한국어로 출력
